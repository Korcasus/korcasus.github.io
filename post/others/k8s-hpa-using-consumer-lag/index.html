Consumer lag을 연동해서 Kubernetes HPA 하기

Kafka Consumer Lag을 Kubernetes Metric으로 연동한뒤, Pod AutoScaling에 활용합니다.

8월 15, 2023
9 분 정도

Required

  • Kubernetes
  • Kafka
  • Burrow, Burrow Exporter
  • Prometheus
  • Helm

Objective

Kafka Consumer lag 값을 연동한 Kubernetes HPA(Horizontal Pod Autoscaling)를 사용하여, Auto Scaling을 하는 것이 목표입니다. auto scaling이 가능해지면 아래 2가지 목표를 달성할 수 있습니다.

  • Consumer lag이 증가하면 Pod을 추가해 처리속도를 늘립니다.
  • Consumer lag이 감소하면 Pod을 줄여 여유 리소스를 확보합니다.

1. Kafka Consumer Lag이란?

producer offset과 consumer offset의 차이가 consumer lag 입니다.

Producer가 partition에 데이터를 기록하면 offset이 증가합니다. Consumer가 partition의 데이터를 처리하면 offset이 증가합니다. Producer offset과 Consumer offset의 차이가 커지면, 데이터를 소비하는데 지연이 있다는 의미로 볼 수 있습니다. 두개의 offset 차이가 적어지면, Producing하는 message가 적다는 것을 의미하거나 Consumer가 필요이상으로 존재한다는 의미로 볼 수 있습니다.

metric 생성을 위한 삽질

외부에서 수집되는 지표이므로, Custom Metric이 아닌 External Metric으로 만들어 주어야한다.

가이드를 참고해서 yaml 파일을 작성한다. (아래에 작성한 yaml을 보셔도 됩니다)

다행히 0.9.0버전 이상에서 cross-namespace를 지원하여, 하나의 namespace에서만 생성해도 다른 namespace에서 사용가능해졌다.

물론 cross-namespace를 적용하여 진행하였다.

관련된 issue를 확인하고싶다면 아래의 링크를 참고면 된다.

HPA 생성

HPA는 생성시 사용자가 지정한 Metric값을 사용하여, 알고리즘을 통해 사용자가 지정한 목표값과 일치하도록 replica 개수를 주기적으로 조정한다.

사용할 Metric은 앞에서 연동한 kafka consumer lag이며, HPA는 이 값을 사용하여 replica 개수를 조정하게 된다.

💡 Custom Control Plane을 만들어 특정조건의 deployment일경우, kafka consumer lag을 사용한 HPA를 자동 생성할 수도 있을것 같다.

Article에서 사용한 hpa.yaml은 아래와 같이 작성되어 있으며, 일반적인 형태이다.

하지만 우리는 다른 namespace에 존재하는 metric을 사용해야하기 때문에, 다르게 작성되어야한다.

참고 용도로 첨부하였다.

kind: HorizontalPodAutoscaler
apiVersion: autoscaling/v2beta1
metadata:
  name: consumer-hpa
spec:
  scaleTargetRef:
    apiVersion: apps/v1
    kind: Deployment
    name: kafka-consumer-application
  minReplicas: 1
  maxReplicas: 10
  metrics:
  - type: Pods
    pods:
      metricName: "kafka_consumergroup_lag"
      targetAverageValue: 5

위의 yaml 파일과 우리가 사용할 yaml 파일의 다른점은 사용해야할 metric 정보를 annotation으로 추가했다는 점이다.

그 외의 정보는 동일하다.

apiVersion: autoscaling/v1
kind: HorizontalPodAutoscaler
metadata:
  name: external-queue-scaler
  # the HPA and scaleTargetRef must exist in a namespace
  namespace: default
  annotations:
    # The "External" metric below targets a metricName that has namespaced=false
    # and this allows the metric to explicitly query a different
    # namespace than that of the HPA and scaleTargetRef
    autoscaling.alpha.kubernetes.io/metrics: |
      [
        {
            "type": "External",
            "external": {
                "metricName": "queue_depth",
                "metricSelector": {
                    "matchLabels": {
                        "namespace": "queue",
                        "name": "my-sample-queue"
                    }
                },
                "targetAverageValue": "50"
            }
        }
      ]
spec:
  maxReplicas: 5
  minReplicas: 1
  scaleTargetRef:
    apiVersion: apps/v1
    kind: Deployment
    name: my-app

Replica 개수 계산 알고리즘

원하는 레플리카 수 = ceil[현재 레플리카 수 * ( 현재 메트릭 값 / 원하는 메트릭 값 )]

  • 원하는 레플리카 수 : HPA를 통해 변경될 replica 수
  • 현재 레플리카 수 : 현재 운영되고있는 replica 수
  • 현재 메트릭 값 : HPA에서 사용할 Metric 값
    • 원하는 메트릭값의 type이 AverageValue일 경우, 현재 메트릭 값 = 측정된 메트릭 값 / 현재 레플리카 수 로 계산된다
  • 원하는 메트릭 값 : HPA의 auto scaling을 통해 Metric이 유지되길 원하는 값
  • ( 현재 메트릭 값 / 원하는 메트릭 값 ) 은 1.0에 아깝고 오차 0.1 이내이면 스케일링을 생략한다. -horizontal-pod-autoscaler-tolerance 플래그를 통해 오차값을 제어할 수 있다.

Ex1

2 = ceil(1 * ( 5 / 3))

  • 현재 레플리카 수 : 1
  • 현재 메트릭 값 : 5
    • 레플리카가 1개이기때문에 5 / 1 = 5
  • 원하는 메트릭 값 : 3
  • 원하는 레플리카 수 : 2

Ex2

4 = ceil(2 * ( 5 / 3 ))

  • 현재 레플리카수 : 2
  • 현재 메트릭 값 : 5
    • 실제 kafka consumer lag = 10 = 현재 메트릭 값 * 현재 레플리카 수
  • 원하는 메트릭 값 : 3
  • 원하는 레플리카 수 : 4

Ex3

2 = ceil(4 * ( 1 / 3 ) )

  • 현재 레플리카수 : 4
  • 현재 메트릭값 : 1
    • 실제 kafka consumer lag = 4 = 현재 메트릭 값 * 현재 레플리카 수
  • 원하는 메트릭 값 : 3
  • 원하는 레플리카 수 : 2

Auto Scale Default Behavior

  • Scale Down
    • 안정화 윈도우 : 300초
    • 정책
      • 퍼센트
        • 15초 주기
        • 100% replica 제거(minimum replica)
  • Scale Up
    • 정책
      • Pod
        • 15초 주기
        • 4개 Pod 추가
      • pod
        • 15초 주기
        • 100% replica 추가

  • https://kubernetes.io/docs/tasks/run-application/horizontal-pod-autoscale/#default-behavior

  • 최대 전송하는 메시지 개수 : 무한
  • 전송할 topic : “hpaTopic”

./kafka-verifiable-producer.sh --topic hpaTopic --max-messages -1 -throughput 3 -bootstrap-server ${server}

결과 정리

  • 초기설정을 잘못되어있어 scale up이 늦게되었다.
  • scale down은 약 5분정도 뒤에 진행되었다. 이는 kubenetes 공식문서에 작성된 기본값인 5분과 유사하다.
  • 정상적으로 된 것을 확인할 수 있다.

Metric Server vs Prometheus-Adapter

→ metric-server : 쿠버네티스 클러스터의 기본적인 지표(pod의 cpu사용량, 메트릭)만 수집한다.

→ prometheus-adapter : resource, custom, external metric에 대해 구현이 되어있으며, k8s 1.6+ 에서는 metric server를 대체가능하다.
